# 小白友好教学法 —— 导师手册

> 把计算机概念翻译成人话的方法论合集

---

## 一、核心原则

### 1.1 "先给类比，再给术语"

坏做法："装饰器是一种高阶函数，它接收一个函数并返回一个新的函数"

好做法："装饰器就像餐厅门口的迎宾员——客人推门进来，迎宾员先拦一下，记个名字，再放人进去。`@app.get('/xxx')` 就是迎宾员，它说'有人访问 /xxx 的时候，叫下面这个函数出来接客'。"

**规则**：每个新概念出现时，先给一个生活里的对应物，再给技术定义。

### 1.2 "每个抽象概念配一个具体例子"

学编程最怕的就是"理论听懂了，一写就废"。解决方法是每个概念都配上**实际运行的输入输出**。

```
例题：curl http://localhost:8000/prescriptions/1
响应（HTTP 200）：
{
  "id": 1,
  "name": "麻黄汤",
  "herbs": [{"name":"麻黄","dosage":"三两"}, ...]
}
```

**规则**：每讲完一个接口/函数，立刻给出 curl 命令 + 实际 JSON 输出。

### 1.3 "先跑通，再理解"

不要从源码讲起。先让学员启动服务、在浏览器里看到真实数据返回——有了感性认识之后再拆解每个零件是干什么的。

顺序：**启动 → 看效果 → 拆代码 → 理解原理**

### 1.4 "零基础假设——每个非常规函数都要解释用法"

学员可能不知道以下内容，每次出现都要配"用法卡片"：

- **Python 内置高阶用法**：列表推导式、元组解包、f-string、类型注解（`str | None`）
- **第三方库函数**：`Query()`、`JSONResponse()`、`fetchone()`、`executemany()`
- **语法糖**：装饰器 `@`、`with` 语句、单元素元组 `(x,)`
- **参数规则**：什么参数是必填、什么可选、传参顺序

**用法卡片模板**：

```
【函数/语法：xxx】
  作用：一句话
  原型：函数签名（哪些参数、哪些可选）
  返回值：返回什么类型、什么形状
  类比：生活化解释
  陷阱：常见的报错原因
  示例：最小可运行代码片段
```

**规则**：第一次出现的非常规函数/语法，必须配用法卡片。第二次出现时可以只给一句话提示。

### 1.5 "类和对象——遇到就解释，不单独开课"

学员可能对 OOP 概念（类、对象、属性、方法）不清晰。不单独开"面向对象编程"课，而是**代码中遇到就顺带解释**。每次解释只需三行：

```
【OOP 小卡片：xxx】
  类 (class) = 饼干模具。对象 (object) = 用模具压出来的饼干。
  属性 = 对象上的设置项（conn.row_factory）。方法 = 对象能做的事（conn.execute()）。
  你已经在用了：name.upper()、nums.append()——只是不知道它叫"方法调用"。
```

**规则**：代码里每次出现 `对象.属性` 或 `对象.方法()`，用一行注释标注它是类/对象/属性/方法中的哪一种。半年后自然内化。

### 1.6 "讲完就要练"

一次课不能只讲不练。理想的节奏是：

| 阶段 | 占比 | 内容 |
|------|------|------|
| 我讲你听 | 30% | 概念 + 类比 + 例题 |
| 你看我写 | 20% | 代码 + 注释 |
| **你亲手写** | **50%** | 变式练习（换个需求让你做） |

**规则**：每学完一个模块，出一个"换皮练习题"——核心逻辑一样，场景稍变——让学员亲手敲。

## 二、类比工具箱

### 2.1 技术 → 生活映射表

| 计算机概念 | 生活类比 | 为什么好用 |
|-----------|---------|-----------|
| API / 后端服务 | 餐厅服务员 | 请求→响应 = 点菜→上菜，404 = "菜卖完了" |
| 路由 @app.get() | 菜单上的菜名 | URL 就是客人点菜用的编号 |
| 路径参数 {id} | 酒店房间号 | "305 号房的客人信息"——精准定位一个 |
| 查询参数 ?category= | 淘宝筛选条件 | "只要红色的、价格100以下的" |
| 数据库连接 | 服务员进后厨 | 每次进→取东西→出来关门（close） |
| 数据库表 | Excel 工作表 | 行=记录，列=字段，表格结构 |
| JOIN | 翻译对照表 | id ←对照表→ 名称，多张表的数据接头 |
| 关联表 | 微信群成员列表 | 一个人→多个群，一个群→多个人 |
| row_factory | 快递单贴标签 | 没标签只能靠位置猜（row[1]），有标签直接看（row["名字"]） |
| fetchone() | 从货架拿一件 | 精准查找一条记录 |
| fetchall() | 搬一整箱 | 查列表、批量返回 |
| commit() | 签字确认 | git commit = 永久保存，不签就白干 |
| 装饰器 @ | 迎宾员/贴标签 | 入口处拦截，注册关系 |
| 类 (class) | 饼干模具 | 模具是模板，不能吃；压出来的饼干才是对象 |
| 对象 (object) | 模具压出的饼干 | `"hello"` 是 str 模具压出的一个具体饼干 |
| 属性 (attribute) | 车上的设置按钮 | `conn.row_factory` = 这辆车的"杯型"旋钮 |
| 方法 (method) | 车能做的事 | `conn.execute()` = 发动车；`name.upper()` = 转大写 |
| import | 工具箱 | 要用什么工具先拿出来 |
| 函数 | 菜谱步骤 | 输入→按步骤做→输出 |
| JSON | 通用菜单格式 | 不管哪家餐厅都看得懂 |

### 2.2 最成功的几个类比（保留复用）

**"关联表 = 微信群成员列表"**（SQL 课）

```
你自己(id) ←→ 成员关系 ←→ 群(id)
方剂(id)  ←→ 关联表    ←→ 药材(id)

"你在哪些群里？" = "甘草出现在哪些方剂里？"
"班群有哪些人？"  = "麻黄汤含哪些药材？"
```

**"Git 三段式 = 购物车结账"**（Git 课）

```
工作区(超市货架) → git add(放进购物车) → git commit(结账付款)
```

**"数据库 commit = 签字确认"**（SQL 课）

```
execute(INSERT) = 填好转账单
commit()       = 签字 → 真正转出去
不 commit      = 单子填了但没签字 → 不算数
```

**"conn.close() = 用完盘子放回厨房"**（FastAPI 课）

每个客人点菜 → 服务员拿新盘子 → 菜端上去 → 盘子收回。不收回 → 盘子堆满 → 无法营业。

## 三、注释风格

### 3.1 文件头：一句话总览 + 类比 + 结构目录

```python
"""
项目名 —— 一句话功能

全文核心类比：API = 餐厅服务员
    请求→响应 = 客人点菜→服务员上菜
    路由 = 菜单

【启动方式】...
【文件结构：读完第1部分再读第2部分...】
"""
```

### 3.2 每个函数：三段式注释

```
def 函数名():
    """
    一句话：这个函数干什么
    类比：像生活中的什么
    例题：输入 → 输出（具体数据）
    """

    # ─── 第1步：xxx ────────────────────
    # 用短注释解释每行
    code...

    # ─── 第2步：xxx ────────────────────
    code...
```

### 3.3 关键行：短注释 + 陷阱标注

```python
(category,)  # ← 逗号不能省！没逗号 Python 不认它是元组
conn.close()  # ← 先关门再 return，不然文件锁泄漏
```

## 四、授课节奏

### 4.1 单次课程结构（90-120分钟）

| 阶段 | 时长 | 内容 |
|------|------|------|
| 回顾 | 5min | 上次学了什么，现在能做什么 |
| 新概念 | 15min | 用类比引入，不写代码 |
| 看代码 | 20min | 我写你看，逐行解释 |
| 跑起来 | 10min | 学员自己跑，看真实效果 |
| **动手** | 30min | 变式练习，学员亲手写 |
| 纠错 | 15min | 讨论卡住的地方 |
| 整理 | 5min | 笔记 + commit + push |

### 4.2 错误处理策略

学员遇到错误时：
1. **不要直接告诉答案**——先问"你觉得哪里出问题了"
2. **给线索，不给答案**——"看看第 47 行少了什么？"
3. **3 分钟原则**——如果卡住超过 3 分钟，给具体提示
4. **错误要记录下来**——放笔记的"踩坑记录"章节

## 五、学后产出标准

每学完一个模块，学员应该有：

| 产出 | 形式 |
|------|------|
| 学习笔记 | 一个 .md 文件，用自己话写的，含踩坑记录 |
| 可运行代码 | 跑得通的 .py 文件 |
| GitHub 提交 | 代码和笔记都推到 GitHub |
| 能独立完成变式 | 换个场景不用帮助能写出来 |

## 六、教学检查清单

每次教学前自查：

- [ ] 这个概念有没有生活类比？学员能一句话说出它是干什么的吗？
- [ ] 每个接口/函数有实际输入输出例子吗？
- [ ] 代码里每个关键行有注释吗？陷阱标注了吗？
- [ ] 教程的最后有"你自己动手"的练习吗？
- [ ] 学员跑过实际效果了吗？（不是看我的输出，是自己敲命令看到输出）
- [ ] 学完有笔记产出吗？
- [ ] `git add → commit → push` 执行了吗？
